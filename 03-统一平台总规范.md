# 统一平台总规范

## 1. 总原则

统一平台后续所有开发，必须遵守以下硬规则：

- 平台只能有一个首页
- 平台只能有一套总导航
- 平台只能有一套登录入口
- 平台只能有一套任务、文件、日志口径
- 模块只能作为平台内部模块存在
- 所有新增和修改都必须进入文档、开发、验收、Git 提交闭环

## 2. 平台固定边界

以下内容一旦确定，后续不得随意漂移：

- 平台名称与定位
- 平台首页角色
- 模块中心结构
- 模块命名方式
- 页面风格基线
- 任务状态体系
- 文件治理方式
- 日志与审计字段口径
- 成功、失败、空状态、加载状态表达

## 3. 统一体验要求

- 用户进入网站后，先看到的是平台，而不是某个独立模块站点
- 用户从任何模块都能返回平台首页
- 模块内部只能有模块级导航，不能重建平台总导航
- 不同模块虽然功能不同，但用户感受必须来自同一个产品

## 4. 统一命名与路径规则

- 模块 key 统一使用 kebab-case
- 模块入口路径统一为 `/modules/<module-key>`
- 平台公共 API 统一为 `/api/platform/*`
- 模块 API 统一为 `/api/modules/<module-key>/*`
- 页面名称优先使用直白中文
- 状态名称与按钮文案全平台统一

## 5. 统一状态规则

平台内所有页面都必须统一处理以下状态：

- 加载中
- 无数据
- 操作成功
- 操作失败
- 权限不足
- 功能未开放
- 处理中

要求：

- 不允许只显示难懂报错
- 必须说明当前发生了什么
- 必须说明下一步可以做什么

## 6. 统一任务规则

平台内所有长流程操作都必须纳入任务模型。

统一任务状态建议使用：

- `queued`
- `running`
- `succeeded`
- `failed`
- `cancelled`

任何模块都不能另起一套完全不同的任务状态命名。

## 7. 统一文件规则

只要涉及上传、导出、生成文件、归档，必须服从平台统一文件治理：

- 文件要有归属模块
- 文件要有创建记录
- 文件要可追踪来源
- 文件要有可见状态
- 文件不能散落在无说明路径

## 8. 统一日志与审计规则

任何模块只要发生以下行为，就必须可追踪：

- 登录与退出
- 创建
- 修改
- 删除
- 导入
- 导出
- 任务发起
- 文件生成
- 批量操作
- 配置变更

至少要留下：

- 谁做的
- 什么时候做的
- 做了什么
- 作用于哪些对象
- 成功还是失败

## 9. 统一功能新增规则

新增功能必须先回答 6 个问题：

1. 它属于平台公共层还是某个模块
2. 它的使用者是谁
3. 它解决什么明确问题
4. 它会不会与已有能力重复
5. 它会不会破坏已有体验
6. 它完成后如何验收

如果这 6 个问题说不清，就不允许直接开做。

## 10. 统一功能删除规则

删除功能前必须写明：

- 删除对象
- 删除原因
- 替代方案
- 影响页面与流程
- 数据如何处理
- 用户如何感知变化

如果没有替代方案或迁移方案，不允许轻易删除成熟功能。

## 11. 新模块准入规则

新模块只有满足以下条件时才允许进入平台：

- 有明确使用场景
- 不是已有模块的重复包装
- 能说清输入、处理、输出
- 能说清与平台公共层的关系
- 能服从统一命名、路径、任务、文件、日志规则
- 能被独立验收

## 12. 禁止事项

- 禁止另起一套首页
- 禁止另起一套导航
- 禁止另起一套视觉体系
- 禁止模块偷偷自建完整站点
- 禁止为了赶进度先做假页面冒充完成
- 禁止没有说明就删功能
- 禁止模块之间各自定义不同规则
- 禁止只开发不写验收标准
