# 模块功能缺口对照表

本文件用于把以下三件事打通：

1. `06-模块应有功能清单.md` 里的目标能力
2. `当前真实状态.md` 里的已完成事实
3. 后续 agent 的实际开发顺序

这不是代码实现文档，而是执行用的缺口地图。

后续任何 agent 如果要继续补产品，必须先读：

1. `ROADMAP.md`
2. `当前真实状态.md`
3. `06-模块应有功能清单.md`
4. 本文件 `07-模块功能缺口对照表.md`

---

## 1. 状态定义

本文件只使用四种状态：

- `已完成`：代码、入口、主流程、验证都已具备
- `部分完成`：已有主干，但缺少关键子能力或验证不足
- `未开始`：当前仓库几乎没有对应能力
- `暂不优先`：属于增强项，当前不作为近期主攻方向

## 4. 近期最值得继续补的功能

以下项是我认为最该继续做的，不建议平均用力。

### 第一批

1. 对象管理详情页补齐“关联任务 / 关联日志 / 关联文件 / 来源批次”
2. 调度模块补“执行历史、失败结果、重试/再执行”
3. 分析模块补“历史任务输入文件清单、结果明细、重跑”
4. 平台通知补“按类型筛选 + 统一回跳对象”
5. 平台任务中心补“失败原因承接 + 标准详情页”

### 第二批

1. 文件中心补统一预览
2. Dashboard 补时间维度切换和热点直达
3. 学习训练补错题重练、继续练习强化
4. 帮助中心补模块级说明与 FAQ

### 第三批

1. 模块收藏 / 最近访问
2. 自定义 Dashboard
3. 搜索高亮、最近搜索
4. 更专业的导出与报告体系

## 5. 推荐执行顺序

后续继续干活时，建议按以下顺序推进：

1. 平台公共承接页补强
   - 任务详情
   - 通知回跳
   - 日志明细
   - 文件预览

2. `record-operations` 深化
   - 让它真正成为统一对象落点

3. `schedule-workbench` 深化
   - 从“能调度”升级到“能运维”

4. `analysis-workbench` 深化
   - 从“能跑分析”升级到“能管理分析结果”

5. `dashboard-workbench` 和 `learning-lab` 做增强
   - 这两个模块当前不缺最小闭环，更适合第二阶段精修

## 6. 后续 agent 写任务时的格式要求

后续任何 agent 开工前，必须明确写出：

- 本次处理模块
- 本次处理能力
- 对应优先级：`P0 / P1 / P2`
- 当前状态：`已完成 / 部分完成 / 未开始`
- 目标状态
- 验证方式

不允许再使用下面这种模糊表述：

- “继续完善一下”
- “顺手优化一些”
- “把模块补完整”

必须改成明确表述，例如：

- “补 `record-operations` 的 P1 详情联动能力：对象详情页展示关联任务、关联日志、关联文件，并补 1 条前端 smoke 断言”
