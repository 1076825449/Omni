# Git 提交规范

## 仓库信息

- GitHub 仓库：`https://github.com/1076825449/Omni.git`
- 默认要求：所有任务开发都以这个仓库为准
- 当前已知：SSH 已配置完成，可直接进行 Git 连接与推送

## 1. 目的

统一平台后续由 AI 持续开发时，必须强制要求每个任务独立提交 Git。

原因：

- 防止任务边界混乱
- 防止做了什么说不清
- 方便回退
- 方便验收
- 方便追踪哪次改动破坏了什么

## 2. 硬规则

- 每完成一个任务包，必须提交一次 Git
- 未提交 Git，不允许继续下一个任务
- 不允许把多个任务混在一个提交里
- 不允许跳过提交说明
- 不允许用模糊提交信息

## 3. AI 每次任务结束后必须执行的 Git 动作

1. 查看本次改动范围
2. 只提交本次任务相关文件
3. 编写清晰提交信息
4. 完成提交后，向你汇报：
   - 提交摘要
   - 影响范围
   - 是否已提交成功

## 4. 提交信息格式

统一采用：

`[任务编号] 动作：结果`

示例：

- `[任务01] 初始化：建立重做统一平台基础结构`
- `[任务04] 骨架：完成平台首页与基础布局`
- `[任务08] 接入：恢复管户分配核心流程`
- `[任务11] 收口：完成统一平台第一轮验收`

## 5. 每次提交前必须检查

- 是否只包含当前任务相关修改
- 是否误改了无关模块
- 是否遗漏重要文件
- 是否包含不该提交的临时文件
- 是否完成了最基本验证

## 6. 每次提交后必须汇报

AI 必须明确写出：

- 本次提交对应哪个任务
- 本次提交做了什么
- 本次提交影响哪些页面或流程
- 还有哪些内容没做

## 7. 分支策略

当前先采用最简单可执行策略：

- 在当前工作分支上按任务连续提交
- 远程仓库以 `git@github.com:1076825449/Omni.git` 为准
- 每完成一个任务，默认推送到当前远程分支，除非用户明确要求先不推送

如果你后续提供更明确的 Git 策略，可升级为：

- 主分支
- 开发分支
- 任务分支
- 合并前验收
