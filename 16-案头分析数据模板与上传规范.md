# 案头分析数据模板与上传规范

本文件用于给后续使用者和 agent 一个统一的上传规范，减少“资料上传了但系统识别偏差”的问题。

后续如果继续增强 `analysis-workbench`，必须同时参考：

1. `ROADMAP.md`
2. `当前真实状态.md`
3. `15-案头分析模块-税务风险专项需求.md`
4. 本文件 `16-案头分析数据模板与上传规范.md`

---

## 1. 当前推荐上传格式

当前稳定推荐：

- `CSV`
- `XLSX`
- `JSON`
- `TXT`
- 图片或 PDF 佐证材料

当前不建议直接上传：

- `XLS`

原因：

- 当前项目已内置最小可用的 `XLSX` 读取能力
- 老旧 `XLS` 格式结构差异较大，暂不作为稳定输入格式

因此，业务使用中应优先把表格资料整理为 `CSV` 或 `XLSX` 后再上传。

如果实际没有表格：

- 有图片或 PDF：上传作为佐证材料
- 没有图片或 PDF：使用页面上的“申报数据手工补录”

当前系统不会把图片/PDF 伪装成已经 OCR 识别的数据。图片/PDF 只作为证据留存和后续核查材料，关键金额仍需要通过补录表进入规则引擎。

---

## 2. 文件命名建议

建议文件名直接体现资料类型与期间。

推荐示例：

- `sales_invoices_2026-03.xlsx`
- `purchase_invoices_2026-03.xlsx`
- `vat_return_2026-03.csv`
- `cit_return_2025.xlsx`
- `financial_statement_2026Q1.xlsx`
- `expense_detail_2026-03.csv`

不推荐：

- `表1.xlsx`
- `最终版.xlsx`
- `企业资料.xlsx`

原因：

- 当前系统会结合文件名和字段名推断资料类型
- 文件名越清晰，自动识别越稳定

## 3. 各资料最低建议字段

### 3.1 销项发票

建议字段：

- `期间`
- `企业名称`
- `纳税人识别号`
- `金额`
- `购方名称`
- `商品名称`

### 3.2 进项发票

建议字段：

- `期间`
- `企业名称`
- `纳税人识别号`
- `金额`
- `供应商名称`
- `商品名称`

### 3.3 增值税申报

建议字段：

- `期间`
- `企业名称`
- `纳税人识别号`
- `申报销售额`
- `申报销项`
- `申报进项`

### 3.4 企业所得税

建议字段：

- `期间`
- `企业名称`
- `纳税人识别号`
- `收入总额`
- `主营业务成本`
- `利润总额`
- `应纳税所得额`

### 3.5 个人所得税

建议字段：

- `期间`
- `企业名称`
- `纳税人识别号`
- `工资薪金`
- `申报人数`
- `个人所得税`
- `应纳税所得额`

### 3.6 财务报表

建议字段：

- `期间`
- `企业名称`
- `纳税人识别号`
- `主营业务收入`
- `主营业务成本`
- `期初存货`
- `期末存货`

### 3.7 费用/报销资料

建议字段：

- `期间`
- `企业名称`
- `纳税人识别号`
- `费用金额`
- `凭证类型`

如为白条、收据、无票报销、情况说明等，应在 `凭证类型` 中清楚体现。

## 4. 字段别名兼容

当前系统已兼容一部分常见字段别名，例如：

- `企业名称` / `纳税人名称` / `公司名称`
- `纳税人识别号` / `税号` / `统一社会信用代码`
- `主营业务收入` / `营业收入` / `收入总额`
- `主营业务成本` / `营业成本` / `成本费用`

但兼容不是无限的。

执行原则：

- 能用标准字段名时，优先用标准字段名
- 同一批数据中，同一字段不要混用多种名称
- 周期字段尽量统一成 `2026-03` 这类格式

## 5. 当前识别规则边界

当前第一阶段能力适合：

- 案头预警
- 风险初筛
- 生成通知书与分析报告初稿

当前还不应夸大为：

- 法律结论自动认定
- 完全替代人工核查
- 行业全覆盖精细判断

因此，后续 agent 若继续增强，应优先做：

- 更完整的字段映射模板库
- 分行业规则阈值
- 上传后字段识别确认页
- 文书版式优化
- 人工复核与修订流程

## 6. 当前上传后反馈

当前系统上传文件后会立即返回：

- `dataset_kind`：系统识别的数据类型
- `source_type`：读取来源类型，例如 `csv`、`xlsx`、`json`、`text`
- `row_count`：读取到的数据行数
- `headers`：标准化后的字段清单
- `required_fields`：该资料类型建议具备的字段
- `missing_required_fields`：当前文件缺少的建议字段
- `warnings`：识别提醒

这一步不是最终分析结论，而是上传质量检查。

如果上传后显示 `未识别` 或提示文本降级读取，应优先调整：

- 文件名是否体现资料类型
- 表头是否使用推荐字段
- 文件是否为空
- 是否把扫描件或非结构化说明文档误当作发票/申报明细上传

上传后的字段确认不是最终结论，而是数据入口质量控制。后续如果缺少建议字段，应优先补充资料或使用手工补录。

## 7. 无表格、无图片时的处理办法

现实业务中，增值税、企业所得税、个人所得税可能没有可导出的表格，也可能没有截图或扫描件。

当前系统采用“关键指标补录”策略：

- 增值税：补录期间、申报销售额、申报销项、申报进项
- 企业所得税：补录期间、收入总额、成本费用、利润总额、应纳税所得额
- 个人所得税：补录期间、工资薪金/劳务报酬、申报人数、个税税额、应纳税所得额

补录数据会保存为本次分析任务的结构化 JSON 文件，进入同一套案头分析规则和文书生成流程。

这不是让用户“随便填”，而是把税务人员已经掌握但无法批量导出的关键申报指标，转为可计算、可追溯、可写入报告的结构化资料。

---

## 8. 风险复核和资料缺口

分析完成后，系统会把每条风险同步为对象管理模块中的一条风险对象。

风险复核状态包括：

- `待核实`
- `已确认`
- `误报`
- `已整改`
- `已移交`

结果页会汇总所有风险要求调取的资料，形成“资料缺口清单”。这份清单用于税务人员下一步核查和通知企业补充资料。
