# 完整信息查询表数据源规范

## 1. 模块定位

`info-query` 是平台级纳税人基础信息数据源，不是单独展示页。它负责导入、查询和复用完整信息查询表，为以下场景提供统一企业画像：

1. 案头分析：按纳税人识别号回填企业名称、行业、属地、主管税务机关、管户部门、税收管理员、信用等级、风险等级。
2. 管户分配：按管户部门、税收管理员、风险等级统计企业分布，作为后续分配和调整依据。
3. 新模块开发：后续涉及企业对象、风险画像、任务派发、核查清单的模块，应优先通过 `info-query` 查询纳税人基础信息。

## 2. 正式入口

- 前端路径：`/modules/info-query`
- 后端前缀：`/api/modules/info-query`
- 模块 key：`info-query`
- 数据模型：`TaxpayerInfo`

## 3. 支持导入格式

当前支持：

1. CSV
2. TSV
3. XLS
4. XLSX
5. JSON

导入接口会按 `owner_id + taxpayer_id` 做更新或新增，同一个用户重复导入相同纳税人识别号时应更新原记录。

## 4. 推荐字段

必填字段：

1. `企业名称`
2. `纳税人识别号`

推荐字段：

1. `法定代表人`
2. `纳税人类型`
3. `登记状态`
4. `行业`
5. `属地`
6. `主管税务机关`
7. `管理分局`
8. `税收管理员`
9. `纳税信用等级`
10. `风险等级`
11. `注册地址`
12. `联系电话`
13. `经营范围`

已兼容常见别名，例如 `税号`、`统一社会信用代码`、`纳税人名称`、`公司名称`、`管户部门`、`管理员`、`信用等级` 等。

## 5. 已实现接口

1. `POST /api/modules/info-query/import`

导入完整信息查询表，返回新增、更新、跳过数量和识别后的表头。

2. `GET /api/modules/info-query/taxpayers`

查询纳税人列表，支持 `q`、`tax_officer`、`manager_department`、`industry`、`region`、`risk_level`、`limit`、`offset`。

3. `GET /api/modules/info-query/taxpayers/{taxpayer_id}`

按纳税人识别号查询单户完整画像。

4. `GET /api/modules/info-query/assignment-stats`

返回按税收管理员、管户部门、风险等级的企业数量统计。

## 6. 案头分析复用规则

案头分析任务详情返回 `taxpayer_profile`。规则如下：

1. 从分析数据中识别 `taxpayer_id`。
2. 使用当前登录用户的 `owner_id + taxpayer_id` 查询 `TaxpayerInfo`。
3. 命中后在任务详情中返回企业画像。
4. 未命中时不阻断分析，`taxpayer_profile` 返回空。

后续增强案头分析时，不应重复维护企业基础字段，应继续复用 `taxpayer_profile`。

## 7. 管户分配复用规则

后续管户分配模块应优先使用以下字段：

1. `tax_officer`：当前税收管理员。
2. `manager_department`：当前管户部门。
3. `risk_level`：风险等级。
4. `industry`：行业。
5. `region`：属地。

调户、批量分配、风险加权分配等能力应基于这张表扩展，不应另建重复的纳税人基础表。

## 8. 验收清单

1. 模块中心存在 `信息查询表` 模块，key 为 `info-query`。
2. 前端可打开 `/modules/info-query`。
3. 可上传 CSV/XLS/XLSX/JSON 信息查询表。
4. 缺少企业名称或纳税人识别号的行被跳过，不污染数据源。
5. 重复导入同一纳税人识别号时更新原记录。
6. 查询页可按企业名称、税号、法人检索。
7. 统计区可展示按管理员、管户部门、风险等级分布。
8. 案头分析中出现同一纳税人识别号时，任务详情返回 `taxpayer_profile`。
9. 数据备份导出包含 `taxpayer_infos`。
10. 后端测试、前端 lint、前端 build、前端 e2e 冒烟测试通过。
