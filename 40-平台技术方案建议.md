# 平台技术方案建议

## 1. 目的

这份文档用于约束后续开发时的技术路线，避免另一个 agent 在没有明确技术边界的情况下随意换栈、乱拆架构、过早复杂化。

这不是唯一技术方案，但它是当前最适合这个全新统一平台项目的默认方案。

## 2. 总体技术原则

- 技术选型以稳定、清晰、易维护为第一优先级
- 优先选择 AI 易于持续开发和修复的技术
- 优先选择一体化平台架构，不要一开始就微服务化
- 优先选择对内部平台友好的技术，而不是追逐新潮
- 公共能力先统一，业务能力后扩展

## 3. 推荐总体架构

建议采用：

- 前端：单一平台前端
- 后端：单一平台后端
- 数据库：单一主数据库
- 文件：统一文件存储目录或统一文件服务
- 模块：在平台内部按模块边界组织

不建议一开始采用：

- 多前端仓库
- 多后端服务
- 微服务拆分
- 事件总线驱动的复杂架构
- 重度中间件依赖

## 4. 前端技术建议

推荐：

- `React`
- `TypeScript`
- `Vite`
- `React Router`
- `Ant Design`

原因：

- 工具链成熟
- AI 对这套栈的生成和修复稳定
- 后续做平台页、表单页、列表页、工作台页效率高
- 对模块化开发友好

### 4.1 前端必须加一层平台组件封装

虽然使用 `Ant Design`，但不能直接满项目裸用组件。

必须建立平台自己的组件层，例如：

- 页面容器
- 平台页头
- 模块页头
- 表格封装
- 表单封装
- 状态组件
- 空状态组件
- 错误状态组件
- 任务状态组件

否则后面页面会再次出现风格漂移。

### 4.2 前端目录建议

建议前端按以下思路组织：

```text
frontend/
  src/
    app/                # 路由、应用入口、布局
    pages/              # 平台公共页面
    modules/            # 模块页面
    components/         # 平台通用组件
    features/           # 跨页面公共能力
    services/           # API 调用封装
    stores/             # 状态管理
    styles/             # 全局样式与主题
    types/              # 类型定义
    utils/              # 工具函数
```

### 4.3 前端状态管理建议

建议使用轻量方案：

- React 自身状态
- 必要时使用 `Zustand`

不建议一开始上：

- 过重的全局状态体系
- 为了“规范”而引入高复杂度状态框架

## 5. 后端技术建议

推荐：

- `Python`
- `FastAPI`
- `SQLAlchemy`
- `Pydantic`

原因：

- 结构清晰
- AI 易于编写和维护
- 文档与接口约束清楚
- 对内部平台、表单型系统、任务型系统、文件处理型系统都合适

### 5.1 后端架构建议

后端应采用单体模块化架构：

- 一个主 API 服务
- 平台公共能力在公共层
- 各业务模块在模块层
- 所有模块共用统一认证、日志、任务、文件体系

不建议一开始拆分为多个独立 API 服务。

### 5.2 后端目录建议

```text
backend/
  app/
    main.py
    core/              # 配置、认证、数据库、异常处理
    platform/          # 平台公共能力
    modules/           # 各业务模块
    schemas/           # 通用 schema
    models/            # 数据模型
    services/          # 平台级服务
    repositories/      # 数据访问
    tasks/             # 任务处理
    files/             # 文件服务
    logs/              # 日志与审计
    utils/             # 工具函数
```

### 5.3 接口设计建议

接口应统一：

- 路由结构
- 响应格式
- 错误格式
- 分页格式
- 权限错误返回

建议统一返回结构：

```json
{
  "success": true,
  "message": "ok",
  "data": {}
}
```

错误时也应保持结构统一，不允许模块各自返回完全不同格式。

## 6. 数据库建议

### 6.1 当前推荐

如果这个平台从一开始就按“正式新项目”建设，优先推荐：

- `PostgreSQL`

原因：

- 比 `SQLite` 更适合平台长期成长
- 更适合多模块、多表、多任务记录
- 更适合后续权限、日志、统计、搜索增强

### 6.2 什么情况下可以先用 SQLite

如果你明确知道：

- 第一阶段只在单机环境使用
- 使用人数很少
- 主要目标是快速跑通平台骨架

那可以先用：

- `SQLite`

但要注意：

- 表结构设计要按未来可迁移到 PostgreSQL 的方式来做
- 不要因为先用 SQLite 就写死很多本地特性

### 6.3 数据表设计建议

平台至少应有这些公共表或公共对象模型：

- 用户
- 角色
- 权限
- 模块注册
- 任务记录
- 文件记录
- 操作日志
- 通知
- 系统配置

模块自己的业务表再按模块扩展。

## 7. 文件与存储建议

平台不应让模块各自随意写文件。

建议统一：

- 上传目录
- 生成文件目录
- 归档目录
- 临时文件目录

并在数据库中记录：

- 文件名
- 文件类型
- 所属模块
- 归属对象
- 创建人
- 创建时间
- 状态

## 8. 认证与权限建议

推荐第一阶段采用：

- `Session + Cookie`

原因：

- 对内部平台最直接
- 易做统一登录
- 易做页面访问控制
- 对同站平台体验更稳定

后续如有需要再扩展：

- 外部身份源
- SSO
- 更复杂的权限矩阵

## 9. 任务机制建议

平台中的长流程必须统一纳入任务体系。

推荐做法：

- 小任务可同步执行，但仍记录任务结果
- 中等或长任务使用后台任务机制
- 所有任务都要可查询状态

不建议一开始就引入复杂任务中间件。  
第一阶段可以用：

- 数据库任务表
- 后端后台任务
- 简单轮询更新

后续再评估更复杂方案。

## 10. 搜索建议

第一阶段：

- 先做结构化搜索
- 搜模块、任务、文件、对象

第二阶段：

- 再做搜索分组、筛选、索引优化

不建议一开始就上复杂全文搜索引擎。

## 11. 测试建议

后续开发必须配套测试，不允许完全靠肉眼判断。

建议：

- 后端：`pytest`
- 前端关键流程：`Playwright`
- 平台关键页面：基础冒烟测试
- 每个任务完成后至少做一次最小验收验证

测试重点不是追求覆盖率数字，而是保住关键流程：

- 登录
- 平台首页
- 模块中心
- 任务中心
- 文件中心
- 日志中心
- 标准模块样板核心流程

## 12. 部署建议

第一阶段建议：

- 本地开发：`Docker Compose`
- 生产或交付：统一启动脚本 + 容器部署

推荐保留：

- 一键启动
- 一键停止
- 一键检查

不建议让每个模块自己单独启动一套服务。

## 13. 项目结构建议

如果采用单仓库，建议结构如下：

```text
Omni/
  docs/                # 产品、规范、任务文档
  frontend/            # 平台前端
  backend/             # 平台后端
  scripts/             # 启动、停止、检查脚本
  data/                # 本地运行数据
  tests/               # 平台级测试
```

如果后续模块很多，也可以演进为：

```text
Omni/
  docs/
  apps/
    web/
    api/
  packages/
    ui/
    shared-types/
  scripts/
  tests/
```

但第一阶段不必过早复杂化。

## 14. AI 开发时的技术约束

必须明确告诉另一个 agent：

- 不得随意换技术栈
- 不得擅自引入高复杂度框架
- 不得过早微服务化
- 不得为单个任务引入全局新基础设施
- 不得绕开平台统一的任务、文件、日志、权限体系

## 15. 当前推荐结论

对于这个全新统一平台项目，当前最推荐的默认技术路线是：

- 前端：`React + TypeScript + Vite + React Router + Ant Design`
- 后端：`FastAPI + Python + SQLAlchemy + Pydantic`
- 数据库：`PostgreSQL` 优先，必要时 `SQLite` 过渡
- 认证：`Session + Cookie`
- 测试：`pytest + Playwright`
- 部署：`Docker Compose + 统一脚本`
- 架构：`单体模块化平台`

一句话总结：

先把它做成一个清晰稳定的单体平台，再考虑复杂扩展，不要一开始就把技术复杂度做得超过产品复杂度。
