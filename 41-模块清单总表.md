# 模块清单总表

## 1. 目的

这份文档用于回答两个问题：

- 这个平台第一批准备做哪些模块
- 每个模块分别解决什么问题

它是模块层的总目录，后续每增加一个模块，都应先在这里登记。

## 2. 当前建议的第一批模块

### 模块 A：分析工作模块

- 模块 key：`analysis-workbench`
- 模块定位：面向“上传资料并生成分析结果”的工作型模块
- 主形态：工作流模块
- 优先级：最高
- 状态：待开发

主要价值：

- 统一处理上传、校验、分析、结果、报告输出
- 适合作为平台里“任务型模块”的样板

### 模块 B：对象管理模块

- 模块 key：`record-operations`
- 模块定位：面向“对象列表、分类、分配、批量操作”的管理型模块
- 主形态：数据管理模块
- 优先级：最高
- 状态：待开发

主要价值：

- 统一处理对象管理、分类、分配、批量调整和导出
- 适合作为平台里“列表型模块”的样板

### 模块 C：学习训练模块

- 模块 key：`learning-lab`
- 模块定位：面向“练习、反馈、复盘、统计”的轻交互模块
- 主形态：轻量交互模块
- 优先级：中
- 状态：待开发

主要价值：

- 统一处理题库、训练、错题、收藏和统计
- 适合作为平台里“沉浸式轻模块”的样板

## 3. 为什么先做这 3 个模块

因为这 3 个模块分别覆盖了平台最重要的 3 种模块形态：

- 工作流型
- 列表管理型
- 轻交互型

如果平台能把这 3 种模块都接好，后续扩展新模块时就不容易失控。

## 4. 当前不建议第一阶段就做的模块

- 复杂知识库模块
- 多人协作编辑模块
- 高度依赖外部系统的接入模块
- 插件市场或开放平台模块
- 高级 BI 分析模块

## 5. 模块登记字段建议

后续新增模块时，至少登记：

- 模块名称
- 模块 key
- 模块目标
- 主形态
- 优先级
- 当前状态
- 是否需要登录
- 是否需要任务体系
- 是否需要文件体系
- 是否需要日志与审计
- 是否已有功能说明文档

## 6. 当前结论

平台第一阶段不应同时开发很多模块。  
建议先围绕这 3 个模块建立稳定样板，再继续扩展。
