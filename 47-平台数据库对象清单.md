# 平台数据库对象清单

## 1. 目的

这份文档用于约束另一个 agent 在实现数据库时不要随意建表、乱命名、重复建模。

目标不是现在就写出所有字段细节，而是先把平台必须有的核心对象模型固定下来。

## 2. 数据库设计总原则

- 平台公共对象优先统一建模
- 模块业务对象在模块边界内扩展
- 命名统一、语义稳定
- 状态字段尽量枚举化
- 日志、任务、文件、权限不要各模块重复造表

## 3. 平台公共对象

以下对象建议作为平台级核心对象：

### 3.1 用户 `users`

用途：

- 平台登录账号
- 平台用户主体

建议核心字段：

- `id`
- `username`
- `display_name`
- `password_hash`
- `status`
- `created_at`
- `updated_at`

### 3.2 角色 `roles`

用途：

- 权限归组

建议核心字段：

- `id`
- `role_key`
- `role_name`
- `description`
- `created_at`
- `updated_at`

### 3.3 用户角色关系 `user_roles`

用途：

- 用户与角色关联

- `id`
- `user_id`
- `role_id`
- `created_at`

### 3.4 权限点 `permissions`

- 页面权限

建议核心字段：

- `id`
- `permission_key`
- `permission_name`
- `scope`
- `description`

### 3.5 角色权限关系 `role_permissions`

- `id`
- `role_id`

## 4. 模块业务对象设计原则

模块自己的对象表建议放在模块命名空间下思考，例如：

- 分析工作模块：分析任务、分析结果、报告对象
- 对象管理模块：业务记录、分类结果、分配结果、规则对象
- 学习训练模块：训练集、练习会话、答题记录、错题、收藏

但这些模块对象不要重复实现平台公共对象。

## 5. 统一字段建议

很多表建议尽量统一这些字段：

- `id`
- `status`
- `created_by`
- `created_at`
- `updated_by`
- `updated_at`

如果涉及删除，优先考虑：

- `is_deleted`
- `deleted_at`
- `deleted_by`

## 6. 状态字段建议

不要让每张表都随意写状态名。

建议：

- 平台级任务状态统一
- 文件状态统一
- 通知状态统一
- 模块自己的业务状态也尽量枚举化

## 7. 关联关系建议

平台内大多数对象建议支持：

- 归属模块
- 归属用户
- 归属业务对象

这样后续搜索、日志、通知、联动都更容易统一。

## 8. 当前强制约束

另一个 agent 在实现数据库前必须遵守：

- 不得绕开平台公共对象自行重建任务表
- 不得绕开平台公共对象自行重建文件表
- 不得绕开平台公共对象自行重建日志表
- 不得让不同模块各自定义完全不同的命名方式

## 9. 当前结论

这份文档不是最终数据库设计稿，但已经足以约束另一个 agent：

- 什么必须是平台公共对象
- 什么应该留在模块边界内
- 什么不能乱建
