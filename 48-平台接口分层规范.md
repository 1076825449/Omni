# 平台接口分层规范

## 1. 目的

这份文档用于约束另一个 agent 在开发 API 时不要把平台接口、模块接口、内部服务接口混在一起。

目标是让接口结构从第一天开始就清楚。

## 2. 接口分层总原则

平台接口建议分为三层：

- 平台公共接口
- 模块业务接口
- 内部服务接口

这三层不能混写。

## 3. 平台公共接口

平台公共接口用于承载全站共享能力。

建议前缀：

- `/api/platform/*`

适合放在这里的内容：

- 登录与会话
- 当前用户信息
- 模块注册中心
- 任务中心
- 文件中心
- 日志中心
- 通知中心
- 系统设置
- 搜索
- 备份恢复

示例：

- `/api/platform/auth/login`
- `/api/platform/auth/logout`
- `/api/platform/me`
- `/api/platform/modules`
- `/api/platform/tasks`
- `/api/platform/files`
- `/api/platform/logs`
- `/api/platform/notifications`
- `/api/platform/settings`
- `/api/platform/search`

## 4. 模块业务接口

模块业务接口用于承载具体模块的业务能力。

建议前缀：

- `/api/modules/<module-key>/*`

示例：

- `/api/modules/analysis-workbench/*`
- `/api/modules/record-operations/*`
- `/api/modules/learning-lab/*`

适合放在这里的内容：

- 模块工作台数据
- 模块列表
- 模块详情
- 模块表单提交
- 模块导入导出
- 模块业务任务

## 5. 内部服务接口

内部服务接口只用于平台内部调用或后端内部调用，不作为公开前端契约。

建议：

- 默认不公开给前端
- 不写进前端直接依赖的契约里

如果确实需要 HTTP 层形式，也应明确标记为内部用途。

## 6. 前端调用规则

前端只能直接依赖两类接口：

- 平台公共接口
- 模块业务接口

前端不应直接依赖：

- 内部服务接口
- 临时调试接口
- 没有纳入文档的私有接口

## 7. 响应结构规范

平台内接口应统一响应结构。

建议：

```json
{
  "success": true,
  "message": "ok",
  "data": {}
}
```

失败时也应保持结构统一，例如：

```json
{
  "success": false,
  "message": "permission denied",
  "data": null
}
```

## 8. 分页结构建议

列表类接口建议统一分页结构。

例如：

```json
{
  "success": true,
  "message": "ok",
  "data": {
    "items": [],
    "total": 0,
    "page": 1,
    "page_size": 20
  }
}
```

## 9. 错误返回规则

错误返回必须清楚，不允许模块各自乱写。

至少应统一这几类：

- 参数错误
- 未登录
- 权限不足
- 对象不存在
- 状态不允许
- 系统异常

## 10. 接口命名建议

接口命名要直白，不要炫技。

推荐：

- 列表：`GET`
- 详情：`GET /{id}`
- 新建：`POST`
- 更新：`PUT` 或 `PATCH`
- 删除：`DELETE`
- 动作类：`POST /{id}/action-name`

## 11. 平台与模块边界规则

以下内容应优先放平台层：

- 登录
- 模块中心
- 搜索
- 任务
- 文件
- 日志
- 通知
- 设置

以下内容应放模块层：

- 模块自己的对象管理
- 模块自己的业务动作
- 模块自己的业务规则
- 模块自己的业务结果

## 12. 任务型接口建议

如果某个动作是长流程，不建议前端直接等待最终结果。

推荐模式：

1. 前端发起动作
2. 后端创建任务
3. 返回任务 id
4. 前端去任务中心或轮询结果

## 13. 文件型接口建议

与文件相关的接口应分开：

- 上传
- 文件记录写入
- 文件列表
- 文件详情
- 文件下载
- 文件归档

不要把上传、处理、记录、下载全塞进一个接口里。

## 14. 搜索型接口建议

搜索应优先走平台公共接口。

模块内部如果有深度搜索，可以再有模块级搜索接口，但必须和平台搜索区分清楚。

## 15. 权限型接口建议

权限判断应尽量在平台统一中间层处理，不要让每个模块都发明一套判断方式。

模块可以补充自己的业务权限，但不能推翻平台基础权限口径。

## 16. 当前强制约束

另一个 agent 在开发接口时必须遵守：

- 不得混淆平台接口和模块接口
- 不得让前端直接依赖内部服务接口
- 不得让不同模块返回完全不同的响应结构
- 不得为了赶进度大量写临时接口而不入文档

## 17. 当前结论

这份文档的作用不是限制开发速度，而是避免后期接口失控。  
只要另一个 agent 严格按这份分层规则开发，平台后面扩展时会轻松很多。
