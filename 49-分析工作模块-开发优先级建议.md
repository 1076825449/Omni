# 分析工作模块开发优先级建议

## 1. 目的

这份文档用于约束另一个 agent 在开发分析工作模块时不要把功能做散，也不要一开始就做过深。

核心原则只有一个：

- 先把“新建任务 -> 上传 -> 校验 -> 发起 -> 结果 -> 导出”做成闭环

## 2. 本模块的主动作

分析工作模块的主动作是：

- 发起一个分析任务，并拿到可查看、可导出的结果

如果这个主动作不成立，其他页面再多都没有意义。

## 3. 第一阶段必须有

以下能力必须先做出来：

- 模块工作台
- 新建分析任务
- 上传资料
- 上传预览与校验
- 分析参数填写
- 发起分析
- 任务进度查看
- 结果页
- 结果摘要
- 结果明细
- 导出结果
- 历史任务列表
- 失败原因展示
- 重试入口

## 4. 第一阶段必须做好

不是“有就行”，而是以下几点必须做顺：

### 4.1 结果页必须分层

结果页必须分为：

- 摘要层
- 明细层

不要把所有结果堆在一个长页面里。

### 4.2 失败反馈必须清楚

分析失败时，必须让用户知道：

- 哪一步失败了
- 为什么失败
- 是否可以重试
- 下一步该做什么

### 4.3 历史任务必须可复看

历史任务页不能只是显示一行记录。  
至少应支持：

- 查看当次结果
- 查看当次输出文件
- 查看状态与错误信息

## 5. 第二阶段再做

这些内容可以后做，不要放进第一轮主闭环里：

- 更复杂的规则配置器
- 更复杂的分析参数模板
- 更复杂的多人协作或审批
- 更复杂的分析对比视图
- 更复杂的智能推荐
- 更复杂的自动归因

## 6. 当前先不要做

以下内容第一阶段明确不要做：

- 高复杂度规则编排器
- 多层级审批流
- 智能自动分析助手
- 模块外自动联动
- 花哨的大屏结果页

## 7. 页面建议

优先级建议：

1. 工作台页
2. 新建任务页
3. 上传与校验页
4. 结果页
5. 历史任务页
6. 导出页

## 8. 验收重点

验收分析工作模块时，重点看这几件事：

- 用户能否独立完成一次分析
- 上传校验是否清楚
- 分析任务状态是否清楚
- 结果是否分层清楚
- 导出是否可用
- 失败是否能定位并重试

## 9. 对另一个 agent 的直接约束

开发本模块时：

- 不要先追求复杂分析能力
- 不要先追求漂亮结果展示
- 不要跳过失败处理
- 不要把历史任务做成只有列表没有内容

一句话：

先把闭环做通，再谈高级能力。
