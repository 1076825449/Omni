# Omni 统一平台最终执行版 Roadmap

## 0. 文档定位

这份文档是当前仓库的唯一执行基线。

后续所有 agent 在开始任何开发、修复、重构、补文档、补测试之前，必须先阅读以下文件：

1. `ROADMAP.md`
2. `最终验收清单.md`
3. `01-网站产品定义.md`
4. `02-网站功能清单.md`
5. `03-统一平台总规范.md`
6. `04-模块设计规范.md`
7. `05-AI开发协作规范.md`
8. 本次要处理的具体任务相关文档

如果 `ROADMAP.md` 与 `README.md`、阶段报告、旧任务包存在冲突，以本文件为准。

## 1. 当前真实判断

当前项目不是“Phase 1-5 已全部最终完成”，而是：

- 已有平台骨架和多模块原型
- 已有较完整的后端接口面
- 已有前端页面和平台框架
- 但仍未完成最终收口

当前未闭环的关键事实：

- 前端 `build` 未通过
- 前端 `lint` 未通过
- 后端 `pytest` 默认执行未通过
- 多处文档宣称完成，但代码和验证结果未完全支撑
- 模块 key、API 前缀、模型字段存在不一致
- 部分功能仍是演示型或占位实现

因此，当前总目标不是继续扩张功能面，而是先完成平台收口，建立可信的“已完成”状态。

## 2. 最终目标

本项目最终目标定义如下，必须同时满足：

1. 平台公共层、模块层、治理层三者一致
2. 所有对外宣称“已完成”的能力都有代码、入口、流程、结果、验证支撑
3. 前后端都能稳定运行
4. 前端 `build` 通过
5. 前端 `lint` 通过
6. 后端测试可运行并通过
7. 至少一套前端冒烟测试可运行并通过
8. 核心主流程可人工验收
9. 文档、代码、测试、验收结论一致

只要上述任一项不满足，就不能对外宣称“最终完成”。

## 3. 总执行原则

后续所有 agent 必须遵守：

1. 先校准事实，再写代码
2. 先修平台基线，再补模块功能
3. 先让质量门通过，再宣布阶段完成
4. 先做闭环任务，再做扩展任务
5. 每次只推进一个明确主题，不并发扩大战线
6. 不允许把占位实现包装成正式完成
7. 不允许文档先宣布完成，代码和验证随后补

## 4. 明确禁止事项

后续 agent 不允许做以下事情：

- 不允许跳过 `ROADMAP.md` 直接开工
- 不允许继续沿用“README 已全部完成”的乐观表述
- 不允许在基线未修复前继续大规模新增模块
- 不允许一边修质量门，一边随意扩展业务范围
- 不允许继续混用 `key` 和 `module_id`
- 不允许继续制造新的硬编码 API 路径和模块命名分叉
- 不允许把模拟任务、写死时间、空配置壳、说明文字恢复流程算作正式交付
- 不允许未写验收说明就声称任务完成

## 5. 执行顺序总览

后续执行统一按以下顺序推进，不允许跳阶段：

1. A 阶段：基线校准
2. B 阶段：一致性修复
3. C 阶段：质量门修复
4. D 阶段：功能闭环修复
5. E 阶段：验收与文档收口

只有当前阶段完成并通过验收，才能进入下一阶段。

## 6. A 阶段：基线校准

### A1. 建立真实状态基线

必须完成：

- 明确哪些功能是“已实现但未验证”
- 明确哪些功能是“已验证可用”
- 明确哪些功能仍是“占位/演示型实现”
- 明确哪些文档结论已经过时

产出要求：

- 更新 `README.md` 的状态表述
- 新增或更新一份“当前真实状态”说明文档
- 给出当前最小可运行方式

验收标准：

- 仓库中不存在“全部完成”但无法验证的表述
- 后续 agent 可以仅通过文档理解当前真实状态

### A2. 明确平台模块总表

必须完成：

- 明确正式模块列表
- 明确每个模块的 key、前端路径、后端 API 前缀、成熟度
- 明确哪些属于平台公共能力，哪些属于独立模块

当前建议正式纳入清单的对象：

- `analysis-workbench`
- `record-operations`
- `learning-lab`
- `dashboard-workbench`
- `schedule-workbench`
- `data-ops` 是否独立成模块，必须明确

验收标准：

- 不再出现同一模块多套命名
- 不再出现前后端路径各写各的情况

## 7. B 阶段：一致性修复

### B1. 模型与字段统一

必须完成：

- `Module` 模型统一使用 `key`
- 清理所有 `module_id` 旧写法
- 清理脚本、接口、导入导出逻辑中的旧字段依赖

重点检查范围：

- `backend/app/models/module.py`
- `backend/app/routers/modules.py`
- `backend/app/routers/data_ops.py`
- `backend/scripts/seed_modules.py`
- `cli.py`

验收标准：

- 搜索代码库时，不再有错误的 `module_id` 业务使用
- 模块注册、导入导出、模块查询使用同一字段体系

### B2. 路径与接口前缀统一

必须完成：

- 模块入口路径统一为 `/modules/<module-key>`
- 模块 API 统一为 `/api/modules/<module-key>/*`
- Dashboard、Schedule 等新增模块必须与规范一致

当前高优先级修复对象：

- `dashboard-workbench`
- `schedule-workbench`

验收标准：

- 前端跳转路径、后端路由前缀、模块注册信息三者一致

### B3. 配置与环境统一

必须完成：

- 消除前端硬编码 API Base URL
- 明确环境变量来源和默认值
- 修正不稳定的默认数据库路径
- 保证本地开发与文档说明一致

验收标准：

- 前后端可通过文档提供的方式启动
- CLI、测试、开发环境不再依赖隐藏前提

## 8. C 阶段：质量门修复

### C1. 前端 Build 修复

必须完成：

- 修复 TypeScript 类型错误
- 修复缺失导入
- 修复接口签名不匹配
- 修复模块页面引用错误

验收标准：

- `frontend` 下 `npm run build` 通过

### C2. 前端 Lint 修复

必须完成：

- 清理未使用变量
- 清理明显的 `any`
- 修复违反当前 ESLint 规则的页面逻辑
- 处理 effect 和状态更新相关问题

验收标准：

- `frontend` 下 `npm run lint` 通过

### C3. 后端测试修复

必须完成：

- 修复 `pytest` 路径与导入问题
- 让现有后端测试可收集、可执行
- 明确标准测试命令

验收标准：

- 后端测试命令可稳定运行
- 至少现有核心接口测试通过

### C4. 冒烟测试修复

必须完成：

- 确认 Playwright 冒烟测试依赖的启动方式
- 保证测试步骤与当前页面结构一致
- 清理显然失效或过于宽松的断言

验收标准：

- 至少一套前端冒烟测试可实际运行

## 9. D 阶段：功能闭环修复

### D1. 平台首页与公共页去演示化

必须完成：

- 替换首页硬编码示例数据
- 让“近期任务”“快捷入口”等基于真实接口
- 统一加载、空状态、错误态表达

验收标准：

- 首页反映真实平台状态，而不是静态演示数据

### D2. 模块闭环补齐

必须完成：

- `schedule-workbench` 前端入口与页面补齐
- 核查各模块是否具备最近任务、历史记录、日志、返回平台入口
- 统一各模块主流程和平台公共能力的连接

验收标准：

- 每个正式模块都满足 `52-模块通用能力要求.md`

### D3. 占位实现收口

必须完成：

- 明确哪些功能保留模拟形态
- 对保留模拟的功能加明确说明
- 能升级为真实闭环的功能优先升级

高优先级对象：

- 分析任务执行逻辑
- Schedule 的 cron/next run 计算
- 模块配置持久化
- 数据导入导出往返验证
- 备份恢复能力边界说明

验收标准：

- 不再把明显演示逻辑当成正式完成项

### D4. WebSocket 与通知闭环

必须完成：

- 如果保留实时通知为正式能力，则前端必须真正消费 WebSocket
- 通知未读数、到达提醒、状态同步要闭环

验收标准：

- “有后端 ws 端点但前端未接入”的状态不能继续存在

## 10. E 阶段：验收与文档收口

### E1. 文档收口

必须完成：

- 更新 `README.md`
- 更新测试运行说明
- 更新模块清单与当前状态
- 补一份最终验收报告

验收标准：

- 文档结论与代码现状一致

### E2. 最终验收

必须完成：

- 逐项执行 `最终验收清单.md`
- 输出每项是否已验证
- 标明剩余风险和未完成项

验收标准：

- 能给出“通过 / 有条件通过 / 不通过”的真实结论

## 11. 每次任务的标准输出格式

后续 agent 每次完成任务后，必须输出以下内容：

1. 本次目标
2. 本次实际改动
3. 影响的页面、接口、脚本或流程
4. 如何验证
5. 哪些旧能力保持不变
6. 仍未解决的问题
7. 是否可进入下一任务

禁止只输出“已完成”或“已修复一批问题”。

## 12. 每次任务的完成定义

一个任务只有在同时满足以下条件时才算完成：

- 代码改动完成
- 文档已同步
- 可运行验证已给出
- 未完成项已如实说明
- 没有引入新的明显命名/路径/边界漂移

## 13. 给后续 agent 的直接指令

后续任何 agent 在开始工作前，必须先执行以下动作：

1. 阅读 `ROADMAP.md`
2. 阅读 `最终验收清单.md`
3. 说明当前要处理的是哪一个阶段、哪一个任务
4. 明确本次不做什么

如果 agent 没有先完成这 4 步，就不应该进入编码阶段。

## 14. 当前建议的下一步

下一位 agent 不应该继续扩展新功能。

下一步建议固定为：

1. 更新 `README.md` 和状态文档，校准真实状态
2. 统一 `Module` 字段与模块路径
3. 修复前端 `build`
4. 修复前端 `lint`
5. 修复后端测试入口

只有完成这 5 项后，才允许进入更深的功能补齐。
