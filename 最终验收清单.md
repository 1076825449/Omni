# Omni 统一平台最终验收清单

## 0. 用途

这份清单用于验收当前仓库是否真的达到“最终可交付状态”。

后续所有 agent 在完成任务后，必须对照本清单说明：

- 哪些项已验证
- 哪些项未验证
- 哪些项不通过
- 是否足以进入下一阶段

如果某项没有实际验证，必须明确写“未验证”，不能默认视为通过。

## 当前已验证结果

截至当前仓库状态，以下命令已经实际跑通：

- `cd frontend && npm run build`
- `cd frontend && npm run lint`
- `cd backend && .venv/bin/pytest ../tests/backend -q`
- `cd frontend && npm run test:e2e`
- `python3 cli.py db status`

这些结果已支撑当前仓库进入“通过”状态；后续 agent 仍需继续按本清单逐项核对平台公共能力、模块闭环、治理能力与文档一致性，避免回归。

## 本轮新增验收重点

- 案头分析结果页每条风险应展示“规则命中明细”，至少说明风险类型、为什么发现、涉及数据、计算过程、判断阈值和建议核实资料。
- 税务事项通知书和税务疑点核实报告导出前应能填写税务机关、文号、联系人、联系电话、整改期限和文书日期，下载文书不得再依赖占位联系人信息。
- 首页应展示“今日应处理”，覆盖今日待核实风险、即将到期整改、逾期未整改和最近新增疑点，并提供查看一户式、记录整改情况、标记已排除、标记已整改操作。
- 逾期口径统一为：最新状态为“整改中”且整改期限早于当前时间；没有整改期限不得判定为逾期。
- 普通业务导航应使用税务人员能理解的入口：首页、查一户企业、案头分析、风险清单、纳税人信息、文书报告、帮助。
- 帮助页应按税务工作路径说明导入信息表、查户、案头分析、查看疑点证据、生成文书、记入台账和跟踪整改。
- 登录页应展示“税务案头助手”品牌；生产模式不得默认展示测试账号密码。
- 一户式工作台应支持按纳税人名称检索；从一户式发起案头分析时，应自动带入纳税人识别号和名称。
- 案头分析资料上传应支持 `.xls`、`.xlsx`、`.csv` 等税务人员常见表格格式。
- 系统设置应提供“文书默认信息”，并在报告导出页自动带出税务机关、联系人、联系电话和默认整改期限。
- 批量标记“整改中”必须填写整改期限和联系人；没有整改期限的整改中记录不得进入正式催办闭环。
- 风险台账导入失败明细必须用行号、税号、失败原因、建议处理方式展示，并支持下载失败清单。

## 1. 基线一致性检查

- `ROADMAP.md` 已存在且内容与当前执行状态一致
- `README.md` 没有夸大“已完成”状态
- 当前阶段目标明确
- 平台模块总表清楚
- 文档、代码、测试结论没有互相打架

判定标准：

- 任意一个新 agent 只看文档就能知道当前真实状态

## 2. 平台统一性检查

- 平台只有一个首页
- 平台只有一套总导航
- 平台只有一套登录入口
- 从任意模块都能明确返回平台
- 不同模块仍然像同一个产品，而不是几个独立站点

判定标准：

- 不存在另起首页、另起导航、另起视觉体系的页面

## 3. 命名与路径规则检查

- 模块 key 全部使用 kebab-case
- 模块入口路径统一为 `/modules/<module-key>`
- 模块 API 统一为 `/api/modules/<module-key>/*`
- 平台公共 API 统一为 `/api/platform/*`
- 模型字段命名一致，没有 `key` / `module_id` 混用

判定标准：

- 前端路由、后端路由、数据库字段、模块注册信息一致

## 4. 启动与环境检查

- 后端可按文档启动
- 前端可按文档启动
- 关键环境变量来源明确
- 没有隐藏依赖导致“文档能看、实际不能跑”
- 本地默认配置合理可用

判定标准：

- 新环境按说明操作后，能进入平台

## 5. 前端质量门检查

- `npm run build` 通过
- `npm run lint` 通过
- 没有明显的页面级报错
- 没有因接口不匹配导致的主流程断裂

判定标准：

- 前端构建产物可生成

## 6. 后端质量门检查

- 后端测试命令明确
- `pytest` 可收集并运行
- 核心认证、任务、文件、通知、搜索接口至少有基础验证
- 关键接口没有明显的字段错位和路由错位

判定标准：

- 后端测试不是“理论可跑”，而是实际可跑

## 7. 冒烟测试检查

- 前端冒烟测试命令明确
- 测试依赖的服务启动方式明确
- 登录流程可测
- 首页可测
- 模块跳转可测

判定标准：

- 冒烟测试能真实发现回归，不是过于宽松的空断言
- 当前已验证：登录、首页、设置页，以及全部 7 个正式模块入口均已纳入 smoke

## 8. 平台公共能力检查

- 登录与会话可用
- 模块中心可用
- 任务中心可用
- 文件中心可用
- 日志中心可用
- 通知中心可用
- 搜索可用
- 统计可用
- 设置页可用
- 帮助与引导入口可用

每项都至少要回答：

- 入口在哪里
- 如何操作
- 成功时看到什么
- 失败时看到什么

- 有返回平台入口

- `record-operations`
- `info-query`
- `risk-ledger`
- `learning-lab`
- `dashboard-workbench`

## 10. 模块通用能力检查

## 12. 数据与治理检查

- 所有长流程操作进入统一任务模型

- 所有关键操作有审计日志
- 登录和退出已进入操作记录；早期 SHA-256 密码会在登录成功后自动升级为带盐 PBKDF2
- 生产环境初始化管理员必须显式提供 `ADMIN_PASSWORD`，不得继续默认使用 `admin123`
- 关键行为的成功/失败可追踪
- 导入导出字段一致
- 备份能力边界明确

- 账号基础信息已明确由管理员维护；个人修改密码已接入真实后端接口，成功后需重新登录并写入操作记录

## 15. 文档与交付检查

- `README.md` 已更新为真实状态
- 测试文档已更新

## 16. 最终判定

最终结论只能选一个：

- 通过
- 有条件通过
- 不通过

### 通过

仅当以下条件同时成立时才可选择：

- 文档一致
- 路径一致
- 质量门通过
- 主流程通过
- 验收可复现

### 有条件通过

适用于：

- 主流程基本可用
- 仍有少量非阻塞问题
- 所有未完成项已明确说明

### 不通过

以下任一情况都应判定为不通过：

- `build` 不通过
- `lint` 不通过
- `pytest` 不可运行
- 主流程断裂
- 文档与事实明显不符
- 存在未说明的占位实现

## 17. 后续 agent 的交付模板

每次任务后必须按以下格式汇报：

1. 本次处理的阶段与任务
2. 本次完成了什么
3. 影响了哪些文件、页面、接口、脚本
4. 我实际验证了什么
5. 哪些项还未验证
6. 哪些风险仍然存在
7. 对照本清单，本次能勾掉哪些项

如果做不到这一点，说明本次任务还不算真正收口。
